## 输入

你将收到以下输入
- `items_details`：一个 JSON 数组。每个对象包含 `label` 与 `reasons` 字段，分别表示待归纳中文条目标签及其在首轮编码中的提取原因/定义。
- `items_pool`：当没有 `items_details` 时，你会收到一个 JSON 字符串数组，数组元素是待归纳中文条目标签。

## 你的任务

1. **利用语义进行规范化与归并**：将含义相近 or 重合的条目标签规范化为简洁的中文维度名称。
   - *注意*：不要仅仅依据标签名称的字面相似性进行归并，请务必深度结合 `items_details` 中每个对象的 `reasons` 来判断它们的真实语义相似性。
2. 将概念上高度相关的条目标签归入中等粒度的维度（dimensions）中，使其在概念上形成具有良好内部一致性的范畴。
3. 保持各个维度边界清晰、互不重叠。通过参考 `items_details` 中的解释，能够帮助你更清晰地界定和撰写维度的 `definition`（定义边界）。
4. 归并并抽象那些提供给你的原始条目标签，而不是强行套用预设代码本。

## 维度归纳约束

你不是在给条目做表面分类，而是在建立一个中等粒度、边界清晰、互斥性较强的扎根编码代码本。

### 维度生成原则

- 维度数量应由输入条目的语义结构自然决定，不预设固定数量。
- 只有当一组条目共享稳定的概念主题，并且无法被已有维度清晰吸收时，才创建新维度。
- 如果一个候选维度只能依赖近义词差异、表述细微差异、评价强度差异或命名习惯差异来成立，不应单独创建。
- 如果两个候选维度之间的边界不能用一句简洁的话说明，应合并为一个更稳健的维度。
- 维度名称必须处在相近抽象层级，避免把很细的行为标签、对象标签和很宽泛的结果性标签并列。
- 不要为了覆盖每个语义细节而过度拆分；代码本应能稳定吸收后续评论。

### 合并优先规则

在创建或保留一个维度前，必须先判断它是否能被其他维度吸收。满足任一条件时应合并：

- 两个维度解释的是同一体验对象，只是评价角度、措辞或抽象程度不同。
- 一个维度只是另一个维度的表现形式、子维度、结果、原因或近义说法。
- 两个维度的定义需要使用高度相近的关键词。
- 一个维度主要依赖宽泛评价词成立，而缺少清晰对象、情境或机制。
- 如果移除两个维度名称后，仅看它们包含的 items 和 definition，无法稳定判断它们应分属不同维度。

### 边界与层级检查

- 每个维度必须有清晰的纳入边界：说明哪些条目属于它。
- 每个维度也必须有清晰的排除边界：说明它不应吸收哪些相邻主题。
- 维度之间应尽量互斥，避免多个维度共同覆盖同一批语义。
- 维度名称应概括其 items 的共同语义，而不是简单拼接其中几个 item 名称。
- 如果某个 item 同时看似属于多个维度，应根据它在输入中的提取原因选择最能解释其语义来源的一个维度。

### 输出前内部自检

在最终输出前，请在内部完成以下检查，但不要把检查过程写入输出：

1. 是否存在两个维度可以互相吸收？
2. 是否存在只靠措辞差异、评价强度差异或抽象层级差异成立的维度？
3. 是否存在结果性维度与原因性维度重复并列？
4. 是否存在定义高度相似但名称不同的维度？
5. 是否存在过宽泛、过细碎或边界不清的维度？
6. 是否存在维度名称只是 items 的机械拼接？
7. 每个 item 是否只出现在一个维度中？
8. 每个 definition 是否能清楚说明该维度与相邻维度的区别？

如果检查发现重复、边界模糊或层级不一致，应主动合并、重命名或调整 items，而不是保留多个相近维度。

## 维度设计规则

- 维度名称可以使用英文学术术语或简洁中文术语，但归入 `items` 的条目标签必须保持为中文，且**必须来自于输入的 `items_details` JSON 数组或 `items_pool` 列表中的原始标签名称**。
- 如果一两个特殊标签可以被更宽泛的维度吸收，就不要为其单独创建很小的维度。
- 不要仅因表述不同就拆分高度相关的条目。请参考它们的提取原因：如果提取原因指向极其类似的行为或感受，可以将它们归为同一个条目标签。
- 不要发明任何输入中不存在的条目标签。
- 每个维度定义都要足够具体，以支持后续的增量编码。维度定义应该详细阐述该维度的语义范围 and 内涵，反映它所包含的各种具体条目的共性。

## 输出要求

只返回结构化输出：

- `dimensions`：对象列表：
  - `name`：维度名称
  - `items`：属于该维度的规范化中文条目标签
  - `definition`：用一句简洁的中文定义该维度的边界

## 质量检查清单

- 同一维度内的条目共享清晰的概念主题。
- 条目名称为中文且简洁。
- 该代码本能够吸收后续评论，而无需频繁改名。
